Ладно, держи “по классам” и “кто за что отвечает”, без лирики и без утопления в каждой запятой. У вас тут маленький заводик: **генератор заказов → общая очередь → процессы-станции → протоколы**.

---

## `MainLoop` (дирижёр этого цирка)

**Роль:** запускает всю систему и держит её живой.

* Создаёт папку `protocols` и файлы логов.
* Настраивает сигналы (`SIGUSR1`) так, чтобы “станции” просыпались, когда пришла новая заявка.
* Делает `fork()` по каждой станции: каждая станция живёт в **отдельном процессе**.
* В родительском процессе запускает поток `Produce()` (производитель заявок).
* В дочерних процессах запускает поток `Consume(station)` (обработчик заявок конкретной станции).

**Методы:**

* `Produce(Params create)`
  Генерирует новые заявки случайного типа топлива и через случайные интервалы. Пытается положить в очередь. Потом шлёт `SIGUSR1` группе процессов, чтобы станции проснулись.
* `Consume(Station station)`
  Ждёт сигнал `SIGUSR1`, потом вынимает из очереди всё подходящее по типу топлива и “обрабатывает” (sleep на случайное время).
* `GetRandomTime(Params)`
  Достаёт случайное время (нормальное распределение) для генерации/обработки.

**Что хранит внутри:**

* `Settings settings_` – параметры системы (станции, время генерации, размер очереди)
* `Queue queue_` – общая очередь (на shared memory)
* `run_` – флаг “работаем/останавливаемся”
* `set_` – набор сигналов, которые ждём (`SIGUSR1`)
* `parent_pid_` – PID родителя, используется как ID группы процессов для `killpg`
* `processes_` – PID всех дочерних процессов (станций)
* `thread_` – поток (в родителе это Produce, в ребёнке это Consume)
* `queue_protocol_` – общий лог очереди

## `Queue` (общая очередь заявок в разделяемой памяти)

**Роль:** хранит заявки так, чтобы их могли видеть **разные процессы**.

* Реализована на **shared memory** + **семафор** для синхронизации.
* По сути: массив `Message` + `current_size` (сколько сейчас лежит).

**Методы:**

* `Push(Message&) -> Message`
  Если очередь переполнена, ставит статус `REJECTED`. Иначе добавляет в конец.
* `Pop(GasType) -> optional<Message>`
  Ищет первое сообщение нужного типа со статусом `EXPECTED`. Помечает как `INWORK`, копирует, удаляет из массива и отдаёт.
* `Remove(...)`
  Сдвигает элементы массива после удаления.

**Почему так:** это простая “очередь” без настоящей кольцевой структуры. Работает, но не супер эффективно и легко ошибиться на границах.

---

## `SharedMemory` (низкоуровневое “где лежит очередь”)

**Роль:** создаёт и управляет сегментом разделяемой памяти (System V IPC).

* Делает `ftok` → `shmget` → `shmat`.
* В начале памяти хранит `int current_size`.
* Дальше хранит массив `Message messages[]`.

**Методы:**

* `OpenSession() -> Session`
  Открывает “сессию” доступа: возвращает объект, который захватывает семафор и даёт указатель на `StoredMessages`.

**Смысл:** Queue вообще не думает про shm-системные вызовы, она работает через `SharedMemory`.

---

## `Session` (RAII-блокировка семафором)

**Роль:** гарантирует, что в shared memory не полезут одновременно два процесса.

* В конструкторе делает `semaphore->Wait()` (вошёл в критическую секцию).
* В деструкторе делает `semaphore->Post()` (вышел).

**Методы:**

* `GetMessages()` возвращает `StoredMessages*` (указатели на size и массив сообщений).

**Смысл:** “зашёл в Session” = “взял lock”. Вышел из области видимости = lock отпустился.

---

## `Semaphore` (обёртка над POSIX именованным семафором)

**Роль:** примитив синхронизации между процессами.

* Внутри `sem_open`, `sem_wait`, `sem_post`, `sem_close`, `sem_unlink`.

**Методы:**

* `Wait()` – занять (если занято, ждать)
* `Post()` – освободить
* `Close()`, `Unlink()` – закрыть и удалить имя

**Смысл:** Shared memory без синхронизации = гонки и развал данных. Семафор это “единственный вход в очередь”.

---

## `Message` / `StoredMessages` (данные)

### `Message`

**Роль:** единица работы системы, “заявка”.

* `id` – уникальный номер
* `gas_type` – какой тип топлива нужен
* `status` – состояние (`EXPECTED`, `INWORK`, `PROCESSED`, `REJECTED`)

Есть `operator<<` чтобы красиво печатать.

### `StoredMessages`

**Роль:** “разметка” shared memory.

* `int* current_size` – сколько элементов
* `Message* messages` – массив

Это не контейнер C++, а просто удобный набор указателей на память.

---

## `Settings` / `Station` / `Params` + `GetSettings` (конфиг системы)

**Роль:** описывает систему из файла `.conf`.

* `Settings`:

  * `stations_params` – список станций
  * `create` – параметры генерации заявок (mean/stddev)
  * `number_message_queue` – вместимость очереди
* `Station`:

  * `id` – номер станции
  * `type` – какой `GasType` она обрабатывает
  * `handle` – параметры времени обработки
* `Params`:

  * `mean`, `stddev` – параметры нормального распределения

`GetSettings(path)` читает конфиг, валидирует и возвращает `optional<Settings>`.

---

## Как оно работает целиком (одним куском)

1. `main` читает настройки → создаёт `MainLoop`.
2. `MainLoop`:

   * поднимает протоколы,
   * блокирует `SIGUSR1` и готовится ждать его через `sigwait`,
   * форкает процессы станций (по каждой `Station`),
   * в родителе запускает producer-поток.
3. Producer:

   * “рожает” `Message`,
   * `queue.Push`,
   * сигналит `killpg(SIGUSR1)` всем станциям.
4. Каждая станция (дочерний процесс):

   * просыпается на `SIGUSR1`,
   * вытаскивает из `queue.Pop(type)` всё подходящее,
   * “обрабатывает” (sleep),
   * пишет протоколы.

Вот. Каждый класс делает одну понятную грязную работу, и вместе они изображают многопроцессную обработку заявок.
